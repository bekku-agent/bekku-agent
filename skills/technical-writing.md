# Technical Writing

Rules and patterns for producing high-quality developer content. Every successful piece reinforces a rule. Every failure adds a correction.

## Voice
- First person as Bekku — technical but approachable
- Never hype. Never fluff. Lead with what the reader can DO.
- Short paragraphs (2-4 sentences). Generous headers. Code early.

## Structure That Works
- **Blog posts (800-1500 words)**: Hook → Problem → Solution with code → Gotchas → Next steps
- **Tutorials (1000-2000 words)**: What you'll build → Prerequisites → Step-by-step with code → Verify → Extend
- **Case studies (600-1000 words)**: Scenario → Approach → Implementation → Results → Takeaway

## Code Examples
- Always specify the language and SDK version
- Use real RevenueCat API calls, not pseudocode
- Show the complete snippet — don't make the reader guess imports
- Include error handling in examples (developers copy-paste)

## What Makes RC Content Good
- Reference actual SDK methods by name (e.g., `Purchases.configure()`, not "initialize the SDK")
- Link to specific doc pages, not just "see the docs"
- Address platform differences when they exist (iOS vs Android gotchas)
- End with a concrete next step, not a vague CTA

## Anti-Patterns
- Don't explain what RevenueCat is in every piece — the audience already knows
- Don't write marketing copy. This is developer advocacy, not sales.
- Don't use placeholder values in code — use realistic examples
